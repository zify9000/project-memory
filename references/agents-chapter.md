## 项目记忆体系

### 冷启动阅读顺序
1. `SOUL.md`（要做什么产品、边界在哪）
2. `ARCHITECTURE.md`（骨架地图）
3. `PROGRESS.md`（进行到哪一步）
4. `design/`（产品长什么样，按需打开对应主题的设计文档）
5. `decision/INDEX.md`（当初怎么拍板的，按需打开具体 ADR）
6. 涉及改动时读对应的 `impl/` 条目
（`decision/archive/` 是冷存储，除非翻历史否则不读）

### 文档维护规则
均为事件触发：箭头左侧的时机一出现就执行，不攒、不等"会话结束"（agent 感知不到那个时刻）。当场漏掉的，由最后一条的收口回看兜底。
- 产品定位、目标用户或边界变化 → 更新 `SOUL.md`（罕见，改了就是大事）
- 对话中拍板了架构取舍 → 当场起草 ADR（状态"草稿（待核对）"）请用户核对，通过后改"已采纳"并登记 `decision/INDEX.md`
- 设计定了或变了 → 原地更新 `design/<主题>.md`（活文档只留当前有效设计，不留历史；为什么这么定的历史在 `decision/`）
- ADR 主题从 `decision/INDEX.md` 主题词表中选一个；登记新主题前先扫词表找近义，复用优先
- 新决策取代旧决策 → 旧 ADR 状态改为"已被 ADR-XXXX 取代"，更新索引
- 合并触发条件满足（同主题 ≥3 条取代链 / 总数 >20 / 里程碑节点）→ 执行合并归档
- 改了某模块的职责/接口/数据流 → 同步更新 `impl/<module>.md`
- 新增/删除目录、改了程序/构建/测试入口 → 同步更新 `ARCHITECTURE.md`（保持一页以内）
- 每次任务收口、准备向用户汇报完成前 → ① 刷新 `PROGRESS.md`（覆写=刷新快照：保留未解决坑、摘除已完成已提交、未合并项标"待合并"）；② 回看本次改动是否漏触发上面任一条规则（尤其建了新目录/改了入口 → `ARCHITECTURE.md`、改了模块 → `impl/`、拍板了 → ADR 草稿、设计变了 → `design/`），漏了当场补上
- 拿不准一个特性该不该做 → 对照 `SOUL.md` 的边界，对不上就先问
- 不确定的信息不要写进文档，宁可留空
