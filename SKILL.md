# 项目记忆结构：初始化与伴随维护

## 核心理念（执行前先内化）

AI agent 的上下文会被压缩和重置，磁盘文档是它唯一的长期记忆。因此项目需要一层"agent 冷启动时先读的东西"。

**根本原则：稳定的、代码推不出来的信息才值得写成文档；代码能回答的问题，让 agent 去问代码。** 

**本 skill 有两个使用时机，不是一次性工具：**
1. **初始化**：拉出整套框架（见"模式一"）。
2. **伴随开发**：此后每次会话中，agent 按规则持续更新各文件（见"模式二"）。初始化完成后，日常维护规则已沉淀在项目的 AGENTS.md 里自主驱动 agent。

## 本 skill 的目录结构

```
SKILL.md                        # 完整方法论（本文件）
references/agents-chapter.md    # AGENTS.md "项目记忆体系"章节的标准副本（唯一事实源，含版本号）
scripts/sync.sh                 # 安装/同步/校验脚本（Linux / macOS）
scripts/sync.ps1                # 同上，Windows PowerShell 版，语义一致
```

## 要创建的结构

```
SOUL.md          # 要做什么产品（最稳定，人写，项目的定位与初心）
design/          # 产品长什么样（稳定，agent 起草 ADR、人核对）
  INDEX.md       # 决策索引：每条决策一行，冷启动时先读它而不是逐个翻文件
  archive/       # 被合并取代的旧 ADR，冷存储，agent 平时不读
impl/            # 产品的内部细节（半稳定，agent 随改动更新）
ARCHITECTURE.md  # 骨架是什么（极薄，一页地图，入口索引用）
PROGRESS.md      # 进行到哪一步（易变，agent 每次会话收尾时更新）
AGENTS.md        # 体系如何维护（"项目记忆体系"章节，由 sync 脚本机械写入）
```

六个文件构成一条完整的追问链：**要做什么产品（SOUL）→ 产品长什么样（design）→ 产品的内部细节（impl）→ 骨架是什么（ARCHITECTURE）→ 现在进行到哪（PROGRESS）→ 怎么维护这套记忆体系（AGENTS）**。

## 与项目其他记忆类文档的分工

这六个文件不是项目里唯一的记忆类文档。总原则：**同一条信息只在一个地方维护，双写必然腐化。** 按边界对象分述：

- **vs README**：README 是给**人看**的灵活大杂烩（安装、上手、贡献指南）；这六个文件是给 **agent 冷启动读**的结构化记忆，各有模板。两者不互抄。
  - **SOUL.md ≠ README 的定位段**。"解决什么问题""产品定位"两者可能都写，但语气和视角不同：README 写给人看（介绍、吸引、说服），SOUL 写给 agent 看，是它做取舍时的裁决依据。SOUL 独有的三样——**边界（明确不做）、成功判据、定位的最终裁决**——才是必须落在 SOUL 里的部分。README 可引用 SOUL，不反向依赖。
  - **ARCHITECTURE.md ≠ README 的目录说明**。它只回答"该去哪找"（入口 + 一句话职责 + 延伸阅读指针），不回答"项目是什么"。
- **vs git log**：git log 由 git 自己维护，不在本体系管辖内；这里只划边界——`PROGRESS.md` 管未完成的活态，已提交的完成项摘除（重复只会腐化）；已提交但**未合并/未发布**的完成项，保留并标"待合并"。
- **vs agent 内置 memory**：内置 memory 通常是会话级私有草稿区；这六个文件是仓库内、共享、可 review 的长期记忆。**不互抄**——草稿区记"本次会话的线索与未定型想法"，当一个想法定型为稳定结论（决策拍板、模块约定固化），它从草稿区"毕业"写进 ADR / impl / SOUL。

## 各文件职责与模板

### SOUL.md —— 要做什么产品（最高位阶）

项目的定位、要解决的问题、服务的对象，以及**明确不做什么**。这是全项目最稳定的文件，也是一切决策的最终标尺：agent 面对"要不要加这个特性""这个重构值不值"这类问题时，答案应回溯到这里。半页以内，写实质，不写口号。

模板：

```markdown
# <项目名> 的初心

## 解决什么问题
<谁，在什么场景下，遇到什么痛。一两句话说透>

## 产品定位
<它是什么、不是什么。和已有方案的本质区别>

## 边界（明确不做的事）
- <不做 X，因为……>
- <不做 Y，因为……>

## 成功的样子
<可检验的判据：什么状态算这个项目成了>
```

其中"边界"一节价值最高——agent 的倾向是有求必应、不断加东西，明确的不做清单是约束它的唯一依据。

### design/ —— 产品长什么样（agent 起草，人核对）

放架构决策、取舍理由、被否决的方案，合起来描述产品被设计成了什么样子、以及为什么是这个样子。这类信息**代码里看不出来，且几乎不随时间变化**。采用 ADR（Architecture Decision Record）形式，一个决策一个文件：`design/NNNN-标题.md`，**编号 `NNNN` 全局递增，合并产生的新 ADR 取下一个全局号**。每条 ADR 的理由都应对齐 SOUL.md——对不齐的决策要么写错理由了，要么决策本身该重做。

#### 产生流程：agent 总结对话，人核对生效

ADR 的素材不在代码里，而在**人与 agent 的对话里**——拍板的那几句话、被否决方案的理由，session 一结束就丢了。所以：

1. **捕捉信号**：对话中出现决策时刻——"就用 X 吧"、"这个不做了"、"A 和 B 选 A"、推翻之前的做法、为一个取舍争论后落定——agent 应当场意识到"这是一条 ADR 素材"。
2. **当场起草**：趁对话上下文还在，agent 把背景、决策、理由、被否决方案整理成 ADR 草稿，状态标 `草稿（待核对）`。
3. **人核对**：把草稿呈给用户确认。用户可能纠正理由、补背景、或直接否掉。**未经核对的 ADR 不得标记为"已采纳"**——agent 自己总结的理由可能是它以为的理由，不是人的理由。
4. **生效登记**：核对通过后状态改"已采纳"，写入 `design/NNNN-标题.md` 并登记 INDEX.md。

单个 ADR 模板：

```markdown
# ADR-0001: <决策标题>

- 日期: YYYY-MM-DD
- 状态: 草稿（待核对）/ 已采纳 / 已被 ADR-XXXX 取代 / 已废弃 / 已合并入 ADR-XXXX
- 主题: <从 INDEX.md 主题词表中选一个>

## 背景
<当时面临的问题和约束>

## 决策
<选择了什么方案>

## 理由
<为什么选它>

## 被否决的方案
<考虑过什么、为什么不选——这部分最容易被代码"说谎"掩盖，务必写>
```

#### 防腐机制：索引 + 状态流转 + 合并归档

"一个决策一个文件"的隐患是文件越积越多、新旧决策互相矛盾，最终没人敢信。用三层机制对抗：

**第一层：索引筛查（`design/INDEX.md`）**。每条决策一行：编号、标题、主题、状态、一句话结论。agent 和人冷启动时只读索引，按需打开具体 ADR——索引本身就是防腐烂的筛查层，让几十条决策的查阅成本保持 O(1)。

**主题词表**（防合并筛查失效的关键）：词表维护在 INDEX.md 顶部，初始化时塞入 8 域默认骨架（可增删，是模板默认不是硬约束）：`架构与边界 / 存储与数据 / 鉴权与安全 / 通信与集成 / 部署与运维 / 前端与交互 / 测试与质量 / 构建与工具链`。每条 ADR 的主题**必须从已登记词表中选一个**（单主题，跨域决策取最主要域、其余在理由里点名）；确需新主题时，先在词表里登记再写 ADR。**登记新主题前必须扫词表找近义，复用优先**——`存储` / `持久化` / `Storage` 三词条并存是合并筛查失效的直接原因。词表是项目级的，不强求通用。

模板：

```markdown
# 决策索引

## 主题词表
架构与边界 / 存储与数据 / 鉴权与安全 / 通信与集成 / 部署与运维 / 前端与交互 / 测试与质量 / 构建与工具链
（新增主题在此登记；登记前先找近义复用）

## 决策
| 编号 | 主题 | 状态 | 结论（一句话） |
|------|------|------|----------------|
| 0001 | 存储与数据 | 已采纳 | 用 SQLite 单文件存储，不引入外部数据库 |
| 0002 | 存储与数据 | 已合并入 0017 | 缓存层放内存，不落盘 |
```

**第二层：状态流转**。新决策取代旧决策时，必须同步做两件事：把旧 ADR 状态改为"已被 ADR-XXXX 取代"（文件保留，历史可追溯），并更新 INDEX.md 对应行。任何 ADR 不许静默删除。

**第三层：合并归档**。触发条件（满足其一即执行）：
- 同一主题的 ADR 累计 ≥ 3 条且存在取代链；
- ADR 总数 > 20；
- 每个里程碑/版本节点做一次例行审查。

合并动作：写一条**新的合并 ADR**，以"当前有效决策"为正文体，把演变历史压缩成一小节；旧 ADR 状态改为"已合并入 ADR-XXXX"并移入 `design/archive/`；INDEX.md 只保留合并后的条目。`archive/` 是冷存储——不删除历史（理由可能翻案），但 agent 日常不读。

### impl/ —— 产品的内部细节

按模板字段写：模块职责、入口、依赖、数据流、关键约定。**颗粒度只到模块级**，不描述函数级细节（那是代码自己的事）。

**模块 = 内聚职责，不是目录**。判据一句话：**"把所有文件重命名后，这文档还大体对吗？"——对就合格**。实践中顶层目录常≈职责，重合时直接用；分叉时以职责为准（一个职责跨多目录就合成一个 impl 文件，不要被目录切碎）。命名用职责名：`impl/auth.md`，不焊目录路径。

**颗粒度健康线（防爆炸）**：单个 impl 文档控制在一屏（~30 行）以内，超了说明是两个职责、拆；少于 5 行的并入相邻模块或干脆不写（代码自己说得清楚的不必立条目）；impl 文件总数 3-8 个健康，**>12 个是"在镜像代码结构"的信号**，退一步合并。

**跨切关注点**（错误处理策略、日志约定、并发模型这类非模块级、但代码推不出来的约定）单列 `impl/conventions.md`，只放项目级约定；一旦它超过一屏就说明该拆回各模块。允许局部腐化，但约定"改完代码顺手更新对应条目"（写进 AGENTS.md 强制）。

模板：

```markdown
# <模块名>

- 职责: <一句话>
- 入口: <文件:行 或 导出的核心接口>
- 依赖: <它依赖哪些模块>
- 数据流: <输入 → 处理 → 输出，几步以内说完>
- 关键约定: <不看代码就不知道的实现约定，如错误处理策略、并发模型>
```

### ARCHITECTURE.md —— 骨架（一页地图）

让 agent 知道"该去哪找"，而不是"那里有什么"。**控制在几十行**，超过就删。模板：

```markdown
# 架构地图

<一段话：这个项目是什么、核心技术栈（展开见 SOUL.md）>

## 目录结构
- `src/foo/` — <一句话职责>
- `src/bar/` — <一句话职责>

## 入口
<程序/构建/测试的入口命令>

## 延伸阅读
- 产品定位与边界见 `SOUL.md`
- 设计决策见 `design/INDEX.md`
- 模块细节见 `impl/`
- 当前工作状态见 `PROGRESS.md`
```

### PROGRESS.md —— 进行到哪一步

agent 上下文重置后最容易丢的是"正在改什么、改到哪了、下一步是什么"。**PROGRESS 是活态快照，覆写不累积**——每次会话收尾"刷新快照"：保留未解决的坑，摘除已提交的完成项（边界细则见"与项目其他记忆类文档的分工"）。

模板：

```markdown
# 工作状态

## 进行中
- <任务A：一句话 + 下一步>
- <任务B：一句话 + 下一步>
（2-4 条；超过说明该收口或拆分）

## 遗留问题 / 坑
<未完成改动的原因、已知坏掉的东西；未合并/未发布的完成项标"待合并">
```

### AGENTS.md —— 体系如何维护

初始化时通过 sync 脚本把"项目记忆体系"章节机械写入项目根的 `AGENTS.md`。标准副本（唯一事实源，含版本号）是 `references/agents-chapter.md`；安装与同步：

```bash
# Linux / macOS:
bash <skill目录>/scripts/sync.sh <项目根目录>          # 安装或同步章节（幂等）
bash <skill目录>/scripts/sync.sh --check <项目根目录>  # 仅校验；缺失/被篡改/过期时退出码 1

# Windows（PowerShell，参数语义相同，-Check 对应 --check）:
powershell -File <skill目录>/scripts/sync.ps1 <项目根目录>
powershell -File <skill目录>/scripts/sync.ps1 -Check <项目根目录>
```

`<skill目录>` 是本 skill 的安装路径——agent 加载 SKILL.md 时即已知。脚本通过自身位置（`BASH_SOURCE` / `$MyInvocation.MyCommand.Path`）定位 `references/`，从任意目录调用均可，不依赖任何环境变量。修改 `references/agents-chapter.md` 后递增 start 标记里的版本号（`vN`），已安装项目重跑 sync 同步。

## 模式一：初始化（拉出框架）

1. **先探查**：读现有代码结构、README、package manifest，弄清技术栈和模块划分。
2. **先写 SOUL.md**：这是唯一必须和用户确认的文件——定位、边界、成功判据不能从代码推断，要直接问，写完请用户过目。其余文件可以基于代码补，这个不行。
3. **创建骨架**：创建 `design/`（含 `INDEX.md`、`archive/`）、`impl/`、`ARCHITECTURE.md`、`PROGRESS.md` 的空骨架。
4. **写 AGENTS.md**：运行 sync 脚本（Linux/macOS 用 `<skill目录>/scripts/sync.sh`，Windows 用 `<skill目录>/scripts/sync.ps1`），把"项目记忆体系"章节机械写入项目根的 `AGENTS.md`（已有文件则追加，不动其他内容）。
5. **填内容，按项目类型分路**：
   - **新建项目**：把初始技术选型起草成第一批 ADR 请用户核对，通过后登记 INDEX.md。
   - **存量项目**：基于真实代码填 `impl/` 各模块条目——拿不准的标注"待确认"或留空，不要写臆测的模块说明；先归并已有的零散决策记录（README 片段、commit message、口头约定）成初始 ADR 集合再进 `design/`，不要直接堆文件；推断不出的设计理由不要编。
6. **收尾**：告诉用户这套结构进入"伴随开发"模式——日常维护由 AGENTS.md 驱动，本 skill 在需要时回来对照。

## 模式二：伴随开发（持续维护）

初始化完成后，日常维护规则已经写死在项目 AGENTS.md 的"项目记忆体系"章节里，agent 每次会话都在读它——本 skill 不复读这些规则，只补充两样东西：**触发时机**和**章节里没有的体检增量**。

**每次会话收尾（轻量同步）**：对照 AGENTS.md 章节的"文档维护规则"逐条过一遍**本次会话产生的**变更——有没有当场拍板却没起草的 ADR、改动的模块是否已同步 `impl/`、`PROGRESS.md` 是否刷新到最新、合并触发条件是否满足。规则本身以章节为准，不在此处重复。

**深度体检（体系健康度审查，不是每次会话都跑）**：

触发分两路，都让人在环里：
- **用户显式喊**（"体检" / "audit memory" / "check 记忆体系"）→ 立即跑全套。
- **agent 检测到强里程碑信号**（版本号字段变更 / 新 git tag / PROGRESS"进行中"清空）→ 主动建议"检测到里程碑信号，建议做一次记忆体检，要我现在跑吗？"，**不自动跑**。不设"每 N 次会话自动体检"——太机械且 agent 数不清会话次数。

体检清单（章节维护规则之外的增量）：
- INDEX.md 与各 ADR 文件的状态是否一致（有无"幽灵条目"或漏登记）；
- 主题词表有无近义重复（如 `存储` / `持久化` 并存）→ 合并词条、同步各 ADR 主题字段；
- 有无**跨会话长期**停留在"草稿（待核对）"的 ADR——提醒用户核对，别让草稿烂在堆里（与收尾查的"本次会话即时漏"是两个时间尺度，不重复）；
- ARCHITECTURE.md 是否超过一页；
- 最近的改动有没有触碰 SOUL.md 的边界——有则提醒用户重新审视；
- AGENTS.md 章节副本是否被篡改或过期 → 运行 `sync.sh --check` / `sync.ps1 -Check` 检测。STALE 时**只报告 + 指引，不自动覆盖**：若因源文件升级 → 用户确认后重跑 sync；若是项目级定制改错了地方（章节是机器领地）→ 指引用户把改动迁到 `references/agents-chapter.md` 后再 sync。

## 反模式（不要做）

只收录正文容易被误读或容易偷懒跳过的点；与正文镜像的细则以正文为准，不在此重复。

- 不要把 agent 起草的 ADR 未经用户核对就标"已采纳"——agent 以为的理由不一定是人的理由。
- 不要在对话结束、上下文丢失后才补 ADR——决策素材在对话里，要当场起草。
- 不要生成"代码现状快照"式文档（逐文件/逐函数的摘要）——它必然腐化，且 agent 读代码比读二手摘要更准。
